title: MCP / AI Integration

Polter exposes an MCP (Model Context Protocol) server so AI tools can discover and execute commands natively. Instead of copy-pasting CLI invocations, your AI assistant can browse available commands, run pipelines, and handle package management (npm, pnpm, yarn, bun) directly.

## Installation

### Automatic

The fastest way to register Polter as an MCP server:

```bash
polter mcp install
```

This adds the server to your **local** Claude Code config (scoped to the current project).

#### Project scope (shared via repo)

```bash
polter mcp install --project
```

Writes to `.mcp.json` in the repository root so every collaborator gets the same setup.

#### Global scope (all projects)

```bash
polter mcp install --global
```

Registers the server in your global Claude Code settings.

### Manual

If you prefer to register manually:

```bash
claude mcp add polter -- ~/.polter/bin/polter-mcp
```

Or add this entry directly to `~/.claude/settings.json`:

```json
{
  "mcpServers": {
    "polter": {
      "command": "~/.polter/bin/polter-mcp"
    }
  }
}
```

## Management

### Update

Re-register the MCP server with the latest version:

```bash
polter mcp update
```

This re-registers the MCP server pointing to your installed binary. Run the install script first to update the binary itself.

| Flag | Effect |
|---|---|
| `--project` | Update the project-scoped `.mcp.json` registration |
| `--global` | Update the global Claude Code registration |

### Status

Check your current MCP server installation:

```bash
polter mcp status
```

Shows the installed version, latest available version, and registration status for each scope (local, project, global).

### Remove

Unregister the MCP server:

```bash
polter mcp remove
```

| Flag | Effect |
|---|---|
| `--project` | Remove the project-scoped registration |
| `--global` | Remove the global registration |

## Available Tools

Once connected, the AI client can call these tools:

| Tool | Description | Input |
|---|---|---|
| `polter_list_commands` | Browse and filter available commands | `tool?` |
| `polter_run_command` | Execute a command by ID | `commandId`, `args?`, `flags?` |
| `polter_status` | Show tool versions and project linkage | — |
| `polter_list_pipelines` | List saved pipelines | — |
| `polter_run_pipeline` | Execute a pipeline by name | `name` |
| `polter_create_pipeline` | Create a new pipeline | `name`, `steps`, `description?`, `source?` |
| `polter_update_pipeline` | Update an existing pipeline | `name`, `description?`, `steps?` |
| `polter_delete_pipeline` | Delete a pipeline | `name` |
| `polter_plan` | Dry-run declarative diff from `polter.yaml` | — |
| `polter_apply` | Apply declarative changes | — |
| `polter_pkg_build` | Run the build script | — |
| `polter_pkg_publish` | Publish to npm registry | `tag?`, `dryRun?` |
| `polter_pkg_install` | Install dependencies | `packages?`, `dev?` |
| `polter_pkg_run_script` | Run a package.json script | `script`, `args?` |
| `polter_pkg_version_bump` | Bump semver version | `type: patch/minor/major` |
| `polter_pkg_audit` | Security audit | `fix?` |
| `polter_pkg_info` | Package registry info | `package` |
| `polter_ps` | List all tracked background processes | — |
| `polter_start` | Start a command as background process | `command`, `args?`, `id?`, `cwd?` |
| `polter_stop` | Stop a background process | `id` |
| `polter_logs` | Read captured output from a process | `id`, `tail?`, `stream?` |
| `polter_find_process` | Find processes by directory | `cwd?`, `filter?`, `tail?` |
| `polter_smart_start` | Start package.json script as background process | `script?`, `cwd?`, `tail?` |
| `polter_run_script_bg` | Run a package.json script in background | `script`, `args?`, `cwd?` |

<Callout type="warn" title="polter_apply executes real commands">
  `polter_apply` runs actual CLI commands against your linked projects (deploys, secret writes, domain changes). These actions are not reversible. Always run `polter_plan` first to review what will change.
</Callout>

## Resources

The MCP server also exposes read-only resources that AI clients can inspect:

| URI | Content |
|---|---|
| `polter://package-json` | Parsed package.json (name, version, deps, scripts) |
| `polter://dependency-tree` | Dependency tree (npm ls --json equivalent) |
| `polter://scripts` | Scripts from package.json |
| `polter://pkg-manager` | Detected manager info (id, version, lockFile) |

## Skill Setup

Polter includes a Claude Code skill that teaches the AI assistant how to use all available tools proactively.

### Install the skill

```bash
polter setup
```

This installs the skill file at `~/.claude/skills/polter/SKILL.md`. The command is idempotent — running it again updates the file if needed, or reports that it's already up to date.

### Activate in Claude Code

Use the `/polter` slash command in Claude Code to activate the skill for the current session. The skill guides the AI to:

- Run `polter_ps` at the start of a session to check for active processes
- Check `polter_logs` after code changes for compilation errors
- Suggest creating pipelines for repetitive workflows

### TUI access

You can also install the skill from the TUI via **System > Skill Setup**.

## Skill Usage Examples

After installing the skill, use `/polter` in Claude Code to activate it. Then ask naturally:

### Process management

```
/polter start the dev server and watch for errors
```

The skill runs `polter_smart_start` to launch `npm run dev` in the background, then periodically checks `polter_logs` for compilation errors or warnings.

```
/polter check what's running and show me any errors
```

Runs `polter_ps` to list active processes, then `polter_logs` on each with `stream: "stderr"` to surface issues.

### Pipelines

```
/polter create a pipeline to build, test, and deploy
```

The skill calls `polter_create_pipeline` with steps for `pkg:build`, `pkg:run_script test`, and `vercel:deploy:prod`, then confirms the pipeline was saved.

```
/polter run the deploy pipeline
```

Executes `polter_run_pipeline` with the named pipeline and streams output.

### Command discovery

```
/polter check what supabase commands are available
```

Calls `polter_list_commands` with `tool: "supabase"` and presents a categorized list of all Supabase CLI commands Polter can run.

```
/polter show me all GitHub commands
```

Calls `polter_list_commands` with `tool: "gh"` to list PR creation, issue management, and other GitHub CLI operations.

### Package management

```
/polter audit dependencies and fix vulnerabilities
```

Runs `polter_pkg_audit` first, then `polter_pkg_audit` with `fix: true` if issues are found. Uses the `polter_audit_resolution` prompt for guidance on manual fixes.

```
/polter bump version, build, and publish to npm
```

Sequences `polter_pkg_version_bump`, `polter_pkg_build`, and `polter_pkg_publish` — using the `polter_version_strategy` and `polter_publish_workflow` prompts to decide the bump type and validate pre-publish checks.

### Declarative state

```
/polter what would change if I apply the polter.yaml?
```

Runs `polter_plan` to show the diff between current infrastructure state and the desired state in `polter.yaml`, without executing anything.

## MCP Tool Usage Examples

When using Polter as an MCP server (without the skill), Claude Code calls tools directly. Here are real workflow examples:

### Monitor after code changes

After editing source files, Claude Code can automatically check for build errors:

```
User: "fix the type error in utils.ts"

Claude Code:
1. Reads and edits utils.ts
2. Calls polter_logs({ id: "npm-run-dev", stream: "stderr", tail: 20 })
3. Verifies the fix compiled cleanly
```

### Multi-step deploy workflow

```
User: "deploy to production"

Claude Code:
1. polter_pkg_build() — runs the build script
2. polter_pkg_run_script({ script: "test" }) — runs tests
3. polter_run_command({ commandId: "vercel:deploy:prod" }) — deploys
```

### Manage background processes

```
User: "start the API and the frontend"

Claude Code:
1. polter_start({ command: "npm", args: ["run", "api"], id: "api" })
2. polter_smart_start({ script: "dev" })
3. polter_ps() — confirms both are running
```

### Create reusable pipelines from patterns

```
User: "I keep running build, test, deploy — save that as a pipeline"

Claude Code:
1. polter_create_pipeline({
     name: "release",
     description: "Build, test, and deploy to production",
     steps: [
       { commandId: "pkg:build" },
       { commandId: "pkg:run_script", args: { script: "test" } },
       { commandId: "vercel:deploy:prod" }
     ]
   })
```

## Supported Clients

- **Claude Code** — full support via `polter mcp install`
- **Cursor** — add the server in Cursor's MCP settings
- Any client that implements the [Model Context Protocol](https://modelcontextprotocol.io)
