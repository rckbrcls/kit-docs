Polter exposes an MCP (Model Context Protocol) server so AI tools can discover and execute commands natively. Instead of copy-pasting CLI invocations, your AI assistant can browse available commands, run pipelines, and handle package management (npm, pnpm, yarn, bun) directly.

## Installation

### Automatic

The fastest way to register Polter as an MCP server:

```bash
polter mcp install
```

This adds the server to your **local** Claude Code config (scoped to the current project).

#### Project scope (shared via repo)

```bash
polter mcp install --project
```

Writes to `.mcp.json` in the repository root so every collaborator gets the same setup.

#### Global scope (all projects)

```bash
polter mcp install --global
```

Registers the server in your global Claude Code settings.

### Manual

If you prefer to register manually:

```bash
claude mcp add polter -- npx -y -p @polterware/polter@latest polter-mcp
```

Or add this entry directly to `~/.claude/settings.json`:

```json
{
  "mcpServers": {
    "polter": {
      "command": "npx",
      "args": ["-y", "-p", "@polterware/polter@latest", "polter-mcp"]
    }
  }
}
```

## Management

### Update

Re-register the MCP server with the latest version:

```bash
polter mcp update
```

This clears the npx cache and re-registers with `@latest`, ensuring you always run the newest release.

| Flag | Effect |
|---|---|
| `--project` | Update the project-scoped `.mcp.json` registration |
| `--global` | Update the global Claude Code registration |

### Status

Check your current MCP server installation:

```bash
polter mcp status
```

Shows the installed version, latest available version on npm, and registration status for each scope (local, project, global).

### Remove

Unregister the MCP server:

```bash
polter mcp remove
```

| Flag | Effect |
|---|---|
| `--project` | Remove the project-scoped registration |
| `--global` | Remove the global registration |

## Available Tools

Once connected, the AI client can call these tools:

| Tool | Description | Input |
|---|---|---|
| `polter_list_commands` | Browse and filter available commands | `tool?` |
| `polter_run_command` | Execute a command by ID | `commandId`, `args?`, `flags?` |
| `polter_status` | Show tool versions and project linkage | ‚Äî |
| `polter_list_pipelines` | List saved pipelines | ‚Äî |
| `polter_run_pipeline` | Execute a pipeline by name | `name` |
| `polter_create_pipeline` | Create a new pipeline | `name`, `steps`, `description?`, `source?` |
| `polter_update_pipeline` | Update an existing pipeline | `name`, `description?`, `steps?` |
| `polter_delete_pipeline` | Delete a pipeline | `name` |
| `polter_plan` | Dry-run declarative diff from `polter.yaml` | ‚Äî |
| `polter_apply` | Apply declarative changes | ‚Äî |
| `polter_pkg_build` | Run the build script | ‚Äî |
| `polter_pkg_publish` | Publish to npm registry | `tag?`, `dryRun?` |
| `polter_pkg_install` | Install dependencies | `packages?`, `dev?` |
| `polter_pkg_run_script` | Run a package.json script | `script`, `args?` |
| `polter_pkg_version_bump` | Bump semver version | `type: patch/minor/major` |
| `polter_pkg_audit` | Security audit | `fix?` |
| `polter_pkg_info` | Package registry info | `package` |
| `polter_ps` | List all tracked background processes | ‚Äî |
| `polter_start` | Start a command as background process | `command`, `args?`, `id?`, `cwd?` |
| `polter_stop` | Stop a background process | `id` |
| `polter_logs` | Read captured output from a process | `id`, `tail?`, `stream?` |
| `polter_find_process` | Find processes by directory | `cwd?`, `filter?`, `tail?` |
| `polter_smart_start` | Start package.json script as background process | `script?`, `cwd?`, `tail?` |
| `polter_run_script_bg` | Run a package.json script in background | `script`, `args?`, `cwd?` |

<Callout type="warn" title="polter_apply executes real commands">
  `polter_apply` runs actual CLI commands against your linked projects (deploys, secret writes, domain changes). These actions are not reversible. Always run `polter_plan` first to review what will change.
</Callout>

## Resources

The MCP server also exposes read-only resources that AI clients can inspect:

| URI | Content |
|---|---|
| `polter://package-json` | Parsed package.json (name, version, deps, scripts) |
| `polter://dependency-tree` | Dependency tree (npm ls --json equivalent) |
| `polter://scripts` | Scripts from package.json |
| `polter://pkg-manager` | Detected manager info (id, version, lockFile) |

## Skill Setup

Polter includes a Claude Code skill that teaches the AI assistant how to use all available tools proactively.

### Install the skill

```bash
polter setup
```

This installs the skill file at `~/.claude/skills/polter/SKILL.md`. The command is idempotent ‚Äî running it again updates the file if needed, or reports that it's already up to date.

### Activate in Claude Code

Use the `/polter` slash command in Claude Code to activate the skill for the current session. The skill guides the AI to:

- Run `polter_ps` at the start of a session to check for active processes
- Check `polter_logs` after code changes for compilation errors
- Suggest creating pipelines for repetitive workflows

### TUI access

You can also install the skill from the TUI via **System > Skill Setup**.

## Supported Clients

- **Claude Code** ‚Äî full support via `polter mcp install`
- **Cursor** ‚Äî add the server in Cursor's MCP settings
- Any client that implements the [Model Context Protocol](https://modelcontextprotocol.io)
